# 日志证据基础层

## 为什么需要

温度校正、温度诊断、功率诊断、安全总览等功能页展示的是诊断结果。结果必须能回到原始日志，否则用户无法确认事件来自哪个日志包、哪个文件、哪一行，以及事件前后发生了什么。日志证据层因此独立于具体诊断功能，也独立于生产作业匹配。

## 基础对象关系

客户 / 区域 / 设备 SN → 日期范围 → 日志包 ZIP → 日志文件 → 原始日志行 → 时间戳 → 诊断事件 → 原始证据。

页面按三层理解：

1. 日志基础层：设备、SN、客户、区域、日期、日志包、日志文件、原始行、证据详情。
2. 功能诊断层：温度诊断、温度校正、功率诊断、热过程分析、安全总览。
3. 结果解释层：命中事件、可信度、作业匹配、入库状态、原始证据跳转、上下文。

## 定位规则

优先使用 `source_file_id + source_file + line_hash` 精确定位。`line_hash` 由 SN、日志包 ID、ZIP 内文件路径、真实行号和原始行计算。

兼容旧事件时按以下顺序回退：

1. 真实行号 `line_no`。
2. 已落库原始行全文匹配。
3. 时间戳附近最近日志行，最大容差 5 分钟。

只有成功找到原始行才返回详情。找不到日志包、ZIP 内文件或目标行时接口明确报错，不显示“已定位”。

## 诊断结果跳转

温度校正结果行传递：

- `sn`
- `date`
- `source_file_id`
- `source_zip`
- `source_file`
- `timestamp`
- `line_hash`
- `line_no`
- `matched_keyword`
- `event_type`
- `from`
- 原查询开始日期、结束日期和关键词

详情状态写入 URL，因此刷新可重新加载。返回温度校正页时恢复原查询条件。

## 事件与作业匹配

诊断事件可以没有匹配作业，但不能没有原始日志证据。详情接口直接读取来源 ZIP，不依赖 `cook_jobs`。若事件时间落在同一日志包的作业时间窗口内，会附带 `matched_job_id`；匹配失败时该字段为空，详情仍可查看。

## 字段说明

| 字段 | 说明 |
| --- | --- |
| `sn` | 设备 SN |
| `customer` | 真实设备档案客户，无来源时为“未知” |
| `region` | 真实设备档案区域，无来源时为“未知” |
| `date` | 日志日期 |
| `timestamp` | 日志行时间戳；旧记录可能来自日志包时间回退 |
| `source_zip` | 来源日志包文件名 |
| `source_file` | ZIP 内来源日志文件路径 |
| `line_no` | ZIP 内文件真实行号，新扫描记录提供 |
| `line_hash` | 原始日志行稳定定位标识 |
| `raw_line` | 原始日志行 |
| `event_type` | 诊断事件类型 |
| `matched_keyword` | 命中关键词 |
| `matched_job_id` | 可选作业 ID |
| `evidence_level` | 高 / 中 / 低；当前真实行号且行内有时间戳为高，使用包时间为中 |

## 接口

### `GET /api/logs/detail`

按设备、日志包、日志文件、时间戳、行号或 `line_hash` 定位原始行，返回 `meta`、`target_line`、前后上下文、来源状态和证据字段。上下文默认前后各 20 行，单侧最多 200 行。

### `GET /api/logs/search-in-file`

在当前 ZIP 内日志文件搜索关键词，返回匹配行号、时间戳、`line_hash` 和原始行。结果可继续进入详情定位。

## 当前限制

- 详情与文件搜索按需下载来源 ZIP，超大包仍受下载大小和网络耗时限制。
- 旧扫描记录没有真实行号；扫描版本 3 会自动重扫补齐。
- 时间戳回退只用于兼容旧记录，不会伪造行号。
- 当前文件搜索是单文件线性扫描，返回最多 500 条。

## 后续扩展

- 缓存已下载 ZIP 或按文件建立只读证据索引。
- 将其他诊断模块统一接入 `line_hash` 跳转。
- 增加跨文件、跨日志包搜索。
- 对日志证据增加存储保留、校验和与访问审计策略。
