## 7. 日志解析口径

入口接口：

- `GET /api/log-analysis/{file_id}`
- `GET /api/admin/log-analysis/{file_id}`

解析流程：

1. 根据 `file_id` 查询 `btyc.machine_ftp`。
2. 取 COS ZIP 地址。
3. 下载 ZIP，默认限制最大 `80MB`。
4. 解析关键文件：
   - `oildrum_board.log`
   - `temperature.log`
   - `main_board.log`
   - `debug*.log`
   - `android*.log`
5. 统计油温/油管温度、每日温度、关键词、关键事件、故障码命中和建议。
6. 缓存到 `/app/cache/log_analysis/{file_id}.json`。

当前猪油桶字段口径来自 `docs/猪油桶相关日志说明 (1).docx`：

- `dev status` 第 1 个数 / 10：油温度。
- `dev status` 第 2 个数 / 10：油管温度。
- `dev status` 第 4 个数：电机速度 RPM。
- `dev status` 第 5 个数：油桶加热 PWM 占空比。
- `pump hall start` 的霍尔数量负数表示投油，正数表示反抽。
- `pump hall finish` 的 `state=1` 表示霍尔缺相检测异常，`state=0` 表示正常。

故障码口径来自 `deploy/data/fault_codes_latest.json`：

- 源文件是 `docs/故障码清单-定位问题看表V3.0.xlsx`。
- 当前抽取 sheet 是“故障码清单V5.0-屏蔽”，共 147 条。
- 支持直接命中错误码。
- 支持 `故障码：1_1_86_F` 这类格式，转换规则为第 1 段 + 第 2 段补两位 + 第 4 段补两位，例如 `1_1_86_F => 1010F`。
- 为减少误报，普通数字错误码只在包含故障/异常/error/fault/errCode 等上下文的行里匹配。

当前还没有做“问题定位结论”，只做结构化呈现和基础分类。

### 7.1 日志包稳定诊断

入口接口：

- `GET /api/log-package-diagnostics/{file_id}`
- 可选参数 `target_job_time`：对某个目标作业时间返回匹配失败原因、前后最近内部作业和建议动作。

核心原则：

- `sop_machinelog` 没有匹配记录，不等于设备没有执行作业。
- 日志包处理顺序是：索引检查 → 下载 → ZIP 校验 → 关键文件识别 → 日志时间范围 → 内部作业片段 → DB 生产记录匹配。
- 内部作业片段优先使用日志原始证据，可识别烹饪、快速养锅、录制菜谱和手动/独立动作。
- ZIP 诊断仍受 `MAX_LOG_ANALYSIS_DOWNLOAD_MB` 限制，不允许无限批量下载。
- 诊断结果缓存于 `/app/cache/log_package_diagnostics/{file_id}.json`，不修改公司生产数据库。

日志包状态口径：

- `已匹配生产记录`
- `DB 未匹配，日志内有作业片段`
- `DB 未匹配，日志内仅有养锅/手动动作`
- `DB 未匹配，日志内无有效作业`
- `日志不可下载`
- `日志不可解压`
- `关键日志缺失`
- `时间范围无法识别`
- `解析失败`

详细字段、失败原因和排查流程见 `docs/LOG_PACKAGE_DIAGNOSTICS.md`。

### 7.2 单设备风险排查工作流

主入口：

- `GET /api/device-risk-overview/{sn}`：汇总近 7/30 天作业、行为需复核数量、日志包诊断覆盖和已解析候选风险。
- `GET /api/thermal-safety-workspace/{sn}`：按 `file_id`、`job_id` 或 `internal_session_id` 读取单次作业的温度、功率、能量、动作与原始证据。
- `POST /api/log-package-diagnostics/{file_id}/clear`：admin 清理单包诊断和温度解析缓存后重新诊断。

页面主路径：

```text
设备风险总览
→ 作业列表
→ 日志证据中心
→ 作业热安全分析
→ 菜谱与资源
```

热安全工作区优先读取 DB 已匹配作业；DB 未匹配但日志内部存在作业片段时，降级展示内部作业，不返回空白页。候选风险标签只用于筛查，不能写成事故结论。

日志包诊断成功缓存 24 小时；下载、解压、解析等失败结果只缓存 15 分钟，避免短暂网络故障长期污染状态。

## 8. 缓存和性能

已有缓存：

- 浏览器 `localStorage`：设备历史和设备报告。
- 后端内存缓存：`REPORT_CACHE`。
- 后端磁盘缓存：`CACHE_DIR` 下的设备报告和日志解析结果。
- 菜谱检索缓存：浏览器 `localStorage` + `CACHE_DIR/recipe_search/*.json`，按维度、关键词、菜谱关键词、时间范围和条数生成缓存 key。
- SQLite 审计库：`/app/cache/zhiku_audit.sqlite3`。

当前默认设备报告缓存 TTL：

- `CACHE_TTL_SECONDS=300`

当前默认菜谱检索缓存 TTL：

- `RECIPE_SEARCH_CACHE_TTL_SECONDS=21600`

性能理解：

- 同 SN 重复查询会优先命中浏览器本地缓存，其次后端缓存。
- 同条件菜谱检索优先命中浏览器本地缓存，其次命中服务端磁盘缓存；刷新按钮会绕过缓存重新查源库。
- 日志 ZIP 下载是 COS 跳转，主要带宽不走本服务器。
- 20 人左右的小团队使用，从设计上可以承受。
- 真正风险在于日志解析大 ZIP、误查大表、API/DB 端口公网暴露、账号密码明文交接。
- 当前 `LOG_ANALYSIS_VERSION=2`，旧版日志解析缓存会被自动跳过并重新解析，避免页面拿到缺少分类、规范、故障码字段的旧结构。

菜谱检索保护策略：

- 客户/设备号维度必须填写关键词。
- 当日/区间使用机器维度允许空关键词，但空关键词最多查 7 天。
- 单次最大时间范围默认 180 天。
- 查询先在小表中模糊匹配设备 SN，再用 `sop_machinelog` 的 `idx_sn_time(sn, create_time)` 查生产日志，避免直接在大表上做客户模糊 join。
- 前端单次展示/导出条数默认 200，后端最大 300。
- 页面提供清设备缓存、清菜谱检索缓存、清全部本地缓存入口。

  - 日志证据中心。
  - 菜谱与资源。
- 旧日志结论、温度诊断、功率诊断和热过程分析继续保留，统一放在“调试入口”。
- 全局上下文条展示当前 SN、客户、地区、版本、日期、数据来源、入库进度、候选风险和最近更新时间。
- 作业列表优先读取云端本地 MySQL 的 `cook_jobs`，接口为 `GET /api/device-jobs/{sn}`；源数据库生产记录保留在下方作为参考。
- 数据读取优先级：
  1. 云端结构化库。
  2. 服务端成功缓存。
  3. 已下载解析结果。
  4. 实时下载 ZIP 并解析。
  5. 公司只读源数据库的受限查询。
- 日志包状态语义：
  - `已入库` 只表示结构化结果已经写入本地分析库。
  - `已匹配 DB 作业` 是独立维度，不能与“已入库”混为一谈。
  - `DB 未匹配但日志内有作业` 不等于未生产。
  - 单包失败不会阻断多包任务，页面要保留失败/跳过备注。
- 失败对象统一包含：
  - `stage`
  - `failure_reason`
  - `message`
  - `suggested_action`
  - `can_retry`
  - `retry_action`
  - `data_available`
- 风险口径：
  - 页面只展示候选风险和待复核证据。
  - 不把系统推断写成事故原因或起火定论。
  - 高温、高功率、高能量、动作窗口和日志缺失需要组合复核。
