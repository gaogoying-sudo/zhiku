# ChatGPT 浏览器协作开发协议

更新时间：2026-06-02 17:26 CST
适用项目：`gaogoying-sudo/zhiku`

本文件用于约束后续在 ChatGPT 浏览器对话中直接讨论、修改、提交本项目代码时的协作方式。目标是避免多个对话、Codex、本地开发、GitHub 之间上下文断裂或互相污染。

---

## 1. 基本结论

后续可以采用以下工作方式：

```text
用户在浏览器里提出需求
  ↓
ChatGPT 先判断需求边界和影响文件
  ↓
必要时读取 docs/SHARED_CONTEXT.md、相关代码和历史文档
  ↓
直接修改 GitHub 仓库中的代码或文档
  ↓
提交后回报改动文件、commit message、验证情况、后续风险
  ↓
同步更新项目内的需求/变更记录，方便 Codex 或其他对话继续接手
```

但每一轮开发必须遵守：

- 不直接改敏感账号文件，除非用户明确要求。
- 不在普通回复、普通文档或 PR 描述中复述明文密码。
- 涉及公司源库查询时必须使用 SN、日期范围或明确过滤条件。
- 不全量同步或全表扫描 `btyc.sop_machinelog`。
- 改动后必须留下工程内记录，不能只停留在聊天记录里。

## 2. 项目公共上下文入口

每个新对话、Codex 会话或其他开发入口，优先读取：

```text
docs/SHARED_CONTEXT.md
```

该文件是项目总上下文，记录：

- 项目定位。
- 当前架构。
- 数据源和关键表。
- 已有功能。
- 缓存和性能策略。
- 用户和安全边界。
- 已完成的重要版本和验证样例。
- 后续优先级建议。

如果某次开发改变了架构、核心数据口径、缓存策略、安全策略、部署方式、重要页面入口，则必须更新 `docs/SHARED_CONTEXT.md`。

---

## 3. 需求和变更记录文件

后续新增需求、变更、争议口径、未完成事项，应优先记录到：

```text
docs/CHANGELOG_REQUESTS.md
```

如果该文件不存在，第一轮相关开发应创建它。

建议记录格式：

```markdown
## YYYY-MM-DD HH:mm CST - 需求标题

### 用户原始意图
- ...

### 本轮决策
- ...

### 改动范围
- 后端：...
- 前端：...
- 文档：...

### 不做事项
- ...

### 验证方式
- ...

### 后续待办
- ...
```

原则：

- 用户的原始口语需求要转成工程可执行语言。
- 不能只记录“做了某功能”，还要记录为什么这么做。
- 如果只是样式小修，可以简记；如果涉及数据口径，必须详细记。

## 4. 每轮开发前的判断

每轮需求开始前，先判断本轮属于哪一类：

### A. 文档/治理变更

例如：补共享上下文、补交接文档、补开发规则。

处理方式：

- 可以只改 `docs/` 或 `handover/`。
- 不部署。
- 回报改动文件和 commit message。

### B. 前端展示变更

例如：页面分栏、筛选、按钮、表格、交互体验。

处理方式：

- 重点检查 `deploy/frontend/index.html`。
- 如涉及接口字段，必须同时检查后端返回结构。
- 回报是否需要部署 web 容器。

### C. 后端数据/接口变更

例如：新增 API、调整查询逻辑、导出 Excel、日志解析。

处理方式：

- 重点检查 `deploy/backend/main.py`。
- 必须确认查询不会扫大表。
- 必须说明缓存策略和异常处理。
- 回报是否需要重启 API 容器。

### D. 部署/安全变更

例如：端口、安全组、账号、密码、Docker Compose、Nginx。

处理方式：

- 必须先读 `handover/02-DEPLOYMENT-AND-RECOVERY.md`。
- 敏感账号只引用文件路径，不复述明文。
- 必须明确回滚方案。

## 5. GitHub 工作方式

默认规则：

- 新功能优先新建分支。
- 小型文档修正可直接提交到 `main`，但必须在回复中说明。
- 大功能不直接合并，优先开 PR，用户确认后再合并。
- 每次提交必须给出 commit message。

推荐分支命名：

```text
feature/<short-topic>
fix/<short-topic>
docs/<short-topic>
chore/<short-topic>
```

推荐 commit message：

```text
feat: add xxx
fix: correct xxx
docs: update xxx context
chore: harden xxx baseline
```

## 6. 敏感信息处理

敏感文件包括但不限于：

```text
deploy/.env
config/db_config.env
handover/01-SECRETS-AND-ACCOUNTS.md
output/zhiku_user_accounts_20260429.csv
```

规则：

- 不主动读取或复述密码，除非用户明确要求处理账号问题。
- 不把密码写进普通文档、代码注释、PR 描述或聊天总结。
- 如果发现敏感信息已进入公开仓库，应提示用户优先轮换密码。
- 后续产品化阶段应逐步从固定账号 `.env` 迁移到数据库账号表或更标准的认证方案。

---

## 7. 数据库和性能红线

红线：

- 不全量查询 `btyc.sop_machinelog`。
- 不对大表做无索引模糊查询。
- 不做全量数据同步。
- 不让普通用户触发大 ZIP 批量解析。
- 不把日志 ZIP 原始文件无限制缓存到服务器。

推荐策略：

- 先查设备/客户/版本等小表，拿到 SN 或 owner 后再查日志表。
- 所有大查询必须带时间范围。
- 宽范围查询使用异步任务或缓存结果。
- 导出 Excel 应限制最大记录数或分阶段生成。
- 重复查询应优先命中浏览器缓存、服务端内存缓存、服务端磁盘缓存。

---

## 8. 每次回复用户时的最小交付格式

完成开发后，回复用户至少包含：

```text
已完成什么
改了哪些文件
是否已提交 GitHub
commit message 是什么
是否已部署/是否需要部署
验证了什么
还有哪些风险或后续待办
```

如果没有实际改代码，也要明确说明：

```text
本轮只完成评估/文档/方案，未改业务代码，未部署。
```

## 9. 当前建议的长期产品化注意事项

如果后续要把智库从内部工具做成稳定产品，应重点补齐：

1. 账号体系：从 `.env` 固定账号升级到数据库账号、角色、禁用、审计。
2. 权限模型：admin、普通用户、只读访客、运维角色分层。
3. 数据查询治理：每个接口写清数据源、索引、范围限制、缓存 TTL。
4. 异步任务：日志解析、Top 榜、宽范围导出不要阻塞 HTTP。
5. 缓存治理：缓存 key、TTL、清理策略、版本号、磁盘占用监控。
6. 导出治理：Excel sheet 口径、最大行数、脱敏策略、导出审计。
7. 日志诊断规则：从结构化展示升级到可解释的问题定位结论。
8. 配置化：资源识别、故障码、日志字段口径从代码中抽出。
9. 部署安全：限制公网 API/DB 端口，SSH 密钥化，密码轮换。
10. 工程拆分：当 `deploy/backend/main.py` 继续变大时，应拆成 routers/services/repositories/schemas。
